Code Reviews: a Cornerstone Practice

8th October 2016

Since moving to Copenhagen earlier this year I've been working for a (large and successful) digital agency. I was shocked to find they didn't do code reviews. I truly (naively) believed that in 2016 every development team had some form of code review process.

It didn't seem like a concious decision either, some of the answers to my questions on the lack of reviews were telling:

The developers on this team are professionals, they don't need their work to be checked
We do code reviews, but only for junior developers during their first 2 weeks
We each have our own areas of the project code, we know our areas best, so code reviews don't make sense for us

Agency Time-Centric Culture

Having worked in a SaaS startup for 4 years prior, the heavy burden of time tracking in agencies came as a surprise. Having to track time to some degree is understandable, since agencies sell time to their clients. And I think some SaaS companies could be more aware of time. But I suspect agency time-centric culture makes it difficult for practices like code reviews to survive.

When you have to log every hour of your day to a client invoice, it's hard to justify spending an hour helping another "consultant" make their code more robust. Project managers love developers who burn through their work fast and have time to spare. Unfortunately they don't have much visibility of the underlying quality of the solutions being built, or the long term effects of poor solutions on a project. I think this should change.

Code Reviews can be a Cornerstone of a Great Team

There are so many benefits to a good code review process that I say they are a cornerstone practice for great teams. I'm not going to prescribe a process, but in my experience it should be one the whole team agrees on, and one which requires everybody to put their code through a review before it gets deployed.

Code reviews improve code quality AND the performance of the team. I'll explain why.

Code Quality

Reduce Bugs: Code reviews help catch bugs early. Bugs are cheaper to fix earlier, so that's great. You will also catch bugs get through your QA process, like subtle memory leaks that slow down your app after it's been running in the browser for an hour.

Quality Incentive: Knowing that your code will be reviewed by someone on your team gives you some extra incentive to do things right and not cut corners.

Prevent Bad Practice: Reviews help keep bad practices out of the code. Bad practices spread, developers love to reuse code and techniques. And the broken window effect means that developers lose respect for poorly maintained code and start to write poor code in response. Code reviews can also reduce existing code smells by stopping their reuse, and encouraging them to be refactored as part of other work.

Quality increases Speed: Why care about code quality, especially in an agency where you just need to get something that works out the door? Well, one of the ways teams can move faster is by having well-engineered and well-maintained code, it makes it quicker to refactor to fit in new features, and it reduces bug counts and fix time. So the project could be delivered faster, with less bugs, and be in great shape for another phase of work.

Developer Engagement

To me this is the most important factor. Any company that build software needs good software engineers, code reviews can help to keep your great developers happy, and help to improve the rest of us.

Learning: When we get comments out of code reviews we're being given new knowlegde, and we're sharing our knowledge with the team through our code. We're helping new developers align with the team, and giving them an opportunity to share fresh skills. Constant learning is a big part of the appeal of being a developer, without it work becomes boring. And bored people quit.

Less Dependency on Individuals: We're reducing dependency on individuals, exposure to each others' work means it's easier to pick up where others left off.

Cohesion: There are opportunities for better team cohesion, discussions on code reviews get people helping each other achieve their objectives

Engagement: Lastly and most importantly, developer engagement is affected by the projects we work on. Good developers care about working on a well-maintained project, having other developers cutting corners and adding poor quality code is only going to make your best developers unhappy. For a software company, having unhappy developers is a gigantic problem.
